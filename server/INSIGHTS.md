## What Works

- **2026-06-27** — To assert that a prompt section (skills/memory/etc.) actually lands in a review, drive a real run through `MockLLMProvider` and read it back from the trace rather than unit-testing the assembler: `POST /pulls/:id/review` → `waitForPrRuns(db, prId, { expected: 1 })` → `GET /runs/:runId/trace` → `prompt_assembly.skills`. Set the agent's `repo_intel:false` so context enrichment is skipped and the assertion is stable. Evidence: `server/test/skills.it.test.ts`.
- **2026-06-27** — For LLM-extraction features (conventions), apply the same "don't trust the model's citation" gate reviewer-core uses for findings: have the model return its `evidence_snippet`, but in the grounding gate verify the file exists under `repo.clonePath` + the cited line range is in-bounds + the model's quoted text actually appears there (normalized whitespace), then RE-DERIVE the stored snippet from the real file lines and DISCARD the model's text. Candidates failing any check are dropped — keep this pure (no I/O) so it's unit-testable without Docker. Evidence: `server/src/modules/conventions/helpers.ts` (`groundCandidates`), mirrors `reviewer-core/src/grounding.ts`.

## What Doesn't Work

- **2026-06-27** — Dependency installs MUST use pnpm, not npm. `npm install <pkg>` fails on this repo: first `ERESOLVE` (pre-existing peer-dep conflict), then even `--legacy-peer-deps` dies with `Cannot read properties of null (reading 'matches')`. `pnpm add <pkg>` works cleanly (a `pnpm-lock.yaml` is the real lockfile; the `npm run …` strings in older notes refer to scripts, not installs). Evidence: `server/pnpm-lock.yaml`, `README.md` (pnpm dev).
- **2026-06-27** — Do NOT apply the "context enrichment is best-effort, swallow & continue" rule to a PRIMARY user action. The conventions `extract` originally wrapped its LLM call in `catch { return [] }`; with no OpenAI key configured this turned a `ConfigError('OPENAI_API_KEY is not configured')` into a `200 []`, so the UI showed a blank "nothing extracted" page with zero clue. Best-effort/silent is only correct when the failed thing is OPTIONAL enrichment of some other deliverable. A user-triggered action whose whole purpose is the result must surface the error (rethrow `AppError`s as-is, wrap unexpected ones in `ExternalServiceError`); reserve `[]` for the genuine "ran fine, found nothing". Evidence: `server/src/modules/conventions/service.ts` (`extract`).
- **2026-06-28** — A path classifier with an "everything else → core" default is a JUNK-DRAWER trap. The Smart Diff `classifyFile` originally enumerated only narrow lock/dist/config/entry patterns and defaulted the rest to `core` — on a real lesson PR (138 files) that put 85 in "Core logic / review closely", led by `docs/…/experiment.md`: 33 `.md`, plus `.claude/` tooling, `CLAUDE.md`, `.gitignore`, `.sh`, `.json`. Fix: explicitly route every non-business-code family OUT of core — docs/markdown/assets (`.md`, `docs/`, root `README/CLAUDE/AGENTS/LICENSE…`, images) → boilerplate; dot-files/dot-dirs (`.claude/`, `.gitignore`, `.husky/`), shell scripts, and `.json` → wiring. Order matters: match boilerplate BEFORE wiring so a `.md` inside a dot-dir (`.claude/skills/x.md`) stays boilerplate, not wiring. After the fix core dropped 85→43 (only `.ts/.tsx`). Lesson: when a bucket is the fall-through default, audit a large real diff's bucket distribution — don't trust the happy-path examples. Evidence: `server/src/modules/pulls/classifier.constants.ts`, `classifier.test.ts`.

## Codebase Patterns

- **2026-06-14** — Shared contracts (`@devdigest/shared`) are vendored as TWO hand-maintained copies — `server/src/vendor/shared/` and `client/src/vendor/shared/` — resolved by tsconfig path alias, NOT auto-synced. Adding a field means editing both in lock-step; the only diffs between copies are comments. Evidence: `server/src/vendor/shared/contracts/trace.ts`, `platform.ts`.
- **2026-06-28** — SUPERSEDES the "only diffs are comments" claim above: the two vendored copies are NOT strictly identical-modulo-comments. `server/src/vendor/shared/contracts/knowledge.ts` exports `AgentVersionConfig` + `AgentVersion` that the `client/` copy OMITS entirely (server-only contracts the client never consumes). So don't assume a diff of the two files is empty when reconciling fields — the client copy is a SUBSET; add a field to the client copy only if the client actually needs it, and expect legitimate server-only exports. Evidence: `diff server/src/vendor/shared/contracts/knowledge.ts client/src/vendor/shared/contracts/knowledge.ts` (AgentVersion block, server-only).
- **2026-06-14** — PR-list per-PR aggregates (score, cost) are computed ON READ in `GET /repos/:id/pulls` via one `inArray` query + JS grouping, never denormalized onto `pull_requests`. "Latest review batch" cost has no batch id in the schema — approximated by summing `agent_runs.cost_usd` within a 120s window of the PR's newest priced run. Evidence: `server/src/modules/pulls/routes.ts`.
- **2026-06-14** — `completeAgentRun`'s `values` shape is declared in TWO places that must match: the repo fn (`repository/run.repo.ts`) AND the interface wrapper (`repository.ts:151`). Adding a field (e.g. `costUsd`) needs both or typecheck fails.
- **2026-06-27** — The backend is MIXED-architecture, not uniformly onion. `agents`/`repos`/`reviews`/`repo-intel` are fully layered (routes→service→repository); `settings`/`pulls`/`polling`/`workspace` are THIN — their `routes.ts` query Drizzle inline (no service/repo). Thin modules are grandfathered as warn-level in `arch:check` (`route-direct-db-legacy`); do NOT copy them — new modules must be layered. Evidence: `server/src/modules/{settings,pulls,polling,workspace}/routes.ts` vs `modules/agents/`.
- **2026-06-27** — `platform/container.ts` is the composition root and is deliberately the ONLY place allowed to import across all layers (adapters + sibling-module repos + module services). No `dependency-cruiser` rule uses `^src/platform` as a `from`, so this is by design — don't "fix" it, and don't copy the pattern into a module. Evidence: `server/.dependency-cruiser.cjs`, `server/src/platform/container.ts:26-31`.
- **2026-06-27** — The "skills" feature is pre-scaffolded everywhere EXCEPT one server wire: tables (`skills`/`skill_versions`/`agent_skills`) + contracts + `agents` link routes/repo + `reviewer-core` `assemblePrompt` (`## Skills / rules` block + `prompt_assembly.skills` trace slot) all already exist. `ReviewInput.skills?: string[]` is plumbed through `reviewPullRequest` but the server NEVER populates it — the single missing injection point is `runOneAgent`, just before the `reviewPullRequest(...)` call. Resolve via `agentsRepo.linkedSkills(agentId)` (already ordered by `order`), filter `skill.enabled` (global master switch), map to bodies. Evidence: `server/src/modules/reviews/run-executor.ts:191`.
- **2026-06-27** — The skills Stats tab needs "agents using this skill" (`GET /skills/:id/agents → Agent[]`). Do NOT reach for `agents/helpers.ts` `toAgentDto` from the skills module — a module→module import trips `arch:check`. Instead do the `agent_skills`→`agents` join inside `SkillsRepository` (a repo reading a sibling TABLE is fine; importing a sibling module's CODE is not) and map rows to `Agent` with a local mapper. Same rule blocked reusing `agents` link-state across modules. Evidence: `server/.dependency-cruiser.cjs`, `server/specs/skills.md` §6.
- **2026-06-27** — The "conventions" feature follows the same pre-scaffolded-but-unwired shape as skills: the `conventions` TABLE (`db/schema/knowledge.ts`), the `ConventionCandidate` vendored contract, a `conventions` entry in the `FeatureModelId` registry (`vendor/shared/contracts/platform.ts` — so the model is chosen via `resolveFeatureModel(container, ws, 'conventions')`, NEVER hardcoded), and even the client i18n (`client/messages/en/conventions.json`) all shipped before any extraction route/service/UI existed. When a lab/lesson names a feature, grep the schema + `FeatureModelId` + i18n first — the rails are usually already laid. Evidence: `server/src/modules/conventions/` (new module wiring the existing rails).
- **2026-06-27** — An agent run is an IMMUTABLE prompt snapshot: `prompt_assembly.skills` in a saved trace reflects the skills linked + enabled AT RUN TIME, and linking/enabling a skill afterwards NEVER back-fills past runs. The executor reads `agents.linkedSkills(agentId)` fresh on each run, filters `skill.enabled`, and joins bodies by `order`. So "I attached a skill but don't see it in the assembly" almost always means the trace predates the attach (re-run to verify), not a broken link — check live link state with `GET /agents/:id/skills` + the skill's `enabled` before suspecting the wiring. Evidence: `server/src/modules/reviews/run-executor.ts:192`.
- **2026-06-27** — `prompt_assembly.skills` in the trace is a SINGLE concatenated STRING, not a structured array: `assemblePrompt` does `parts.skills.join('\n\n')` under one `## Skills / rules` header (`reviewer-core/src/prompt.ts:88,109`), and `ReviewInput.skills` is just `string[]` (bodies, no names). So per-skill labels/delimiters CANNOT be added in the client trace viewer (the boundaries are unrecoverable from the flattened text) — they must be embedded at assembly time. Done in `run-executor` by mapping each enabled linked skill to `### ${skill.name}\n\n${skill.body}` before the join (the engine contract stays `string[]`); this labels skills for BOTH the model and the trace view. Evidence: `server/src/modules/reviews/run-executor.ts:193`.

- **2026-06-28** — To let one module's flow reuse ANOTHER module's SERVICE without tripping the `no-cross-module-internals` arch rule, expose the service on the container (the composition root is exempt — no rule uses `^src/platform` as a `from`) and consume it via `this.container.<svc>` — a property access, NOT a static import edge. The rule's `to:` only matches sibling `service`/`repository` FILES, so importing a non-service helper like `settings/feature-models.ts` is fine, but `new ReviewRepository()` from another module is not — read shared data through the already-exposed `container.reviewRepo`. This is how the Intent Layer wires into the reviews executor (intent computed ONCE per PR in `executeRuns`, not per agent). Evidence: `server/src/platform/container.ts` (`get intent()`), `server/src/modules/intent/service.ts` (uses `container.reviewRepo`, never `new ReviewRepository`), `server/src/modules/reviews/run-executor.ts` (`buildIntent` → `container.intent`).

- **2026-06-28** — Smart Diff (`GET /pulls/:id/smart-diff`) is a DETERMINISTIC compose with NO model call — the expensive LLM work already happened in the Structured Reviewer. It classifies each changed file PATH-ONLY into core/wiring/boilerplate (`classifier.ts`, precedence boilerplate→wiring→core so a generated `dist/index.js` beats the `index.*` wiring rule) and overlays the LATEST review's finding lines, computed ON READ — it does NOT persist to `pr_brief`, and `SmartDiffFile.pseudocode_summary` stays `null` (that's the LLM field, intentionally unfilled this lesson). Added inline in the thin `pulls` module (consistent with its grandfathered `route-direct-db-legacy` warn, not a new layered module). Patterns + the `BIG_PR_LINE_THRESHOLD` split heuristic live in `classifier.constants.ts`. Evidence: `server/src/modules/pulls/{classifier,classifier.constants,smart-diff,routes}.ts`.

## Tool & Library Notes

- **2026-06-14** — New DB columns: edit `db/schema/*.ts`, then `npm run db:generate` (drizzle-kit) auto-generates `00NN_*.sql` (e.g. `0010_solid_baron_zemo.sql` = `ALTER TABLE … ADD COLUMN`). Never hand-write migration SQL; apply with `npm run db:migrate`.
- **2026-06-27** — Widening a Drizzle `text({ enum: [...] })` column generates NO migration — the DB type stays `text`, the enum list is a TS-only compile-time narrowing. So adding e.g. `'imported_file'` to `skills.source` needs only the schema edit + both vendored `SkillSource` contracts in lock-step; `db:generate` produces nothing. (Contrast a real pg `enum` type, which WOULD need an `ALTER TYPE`.) Evidence: `server/src/db/schema/skills.ts`.
- **2026-06-27** — Onion layering is machine-enforced: `pnpm arch:check` runs `dependency-cruiser` against `server/.dependency-cruiser.cjs` (also an `arch` job in `.github/workflows/server-unit.yml`). Error-severity rules fail CI; 6 known warns are grandfathered debt (4 thin-module routes + 2 repo-intel service→adapter) — keep errors at 0, don't grow the warns. Config sets `tsPreCompilationDeps: false` ON PURPOSE: counting `import type` edges flags harmless cycles (helpers⇄repository on a row type; container⇄service DI root). `dependency-cruiser` was already a dep (repo-intel indexer) — reused, not added. Evidence: `server/.dependency-cruiser.cjs`.
- **2026-06-27** — `container.repoIntel.getConventionSamples(repoId, n)` returns only top-N ranked file PATHS (tests/configs/migrations stripped), NOT file contents and NOT config files — despite the "conventions sample" name implying configs + code. To build an extraction sample you must read the paths' contents yourself via `readFile(join(repo.clonePath, relPath),'utf8')` (the indexer pattern at `repo-intel/pipeline/incremental.ts:154`) AND add config files (`tsconfig.json`, `.eslintrc*`, `.prettierrc*`, `package.json`) separately. Evidence: `server/src/modules/repo-intel/service.ts:630` (delegates to `getTopFilesByRank`).
- **2026-06-27** — Provider API keys are NOT (only) in `server/.env` — `LocalSecretsProvider` reads `~/.devdigest/secrets.json` (UI "BYO key" writes, mode 0600) FIRST, falling back to `process.env`. `container.llm(provider)` throws `ConfigError('<PROVIDER>_API_KEY is not configured')` when that provider's key is absent in both. Gotcha for any system LLM feature: a `FeatureModelId`'s `defaultProvider` may differ from the providers the user actually has keys for (e.g. `conventions` defaults to `openai/gpt-5.4`, but a user with only Anthropic+OpenRouter keys gets a hard ConfigError until they pick a different model in Settings → Feature Models). To debug "which keys exist", inspect `~/.devdigest/secrets.json`, not `.env`. Evidence: `server/src/adapters/secrets/local.ts`, `server/src/platform/container.ts:175-191`, `vendor/shared/contracts/platform.ts` (FEATURE_MODELS defaults).
