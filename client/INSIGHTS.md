## What Works

- **2026-06-14** ‚Äî `formatCost` (`src/lib/cost.ts`) distinguishes MISSING data (`null`/`undefined` ‚Üí "‚Äî") from a genuine zero (`0` ‚Üí "$0.00"), widens precision for sub-cent values (~2 sig figs), and trims trailing zeros to a 2dp floor ("$0.06" not "$0.060", "$0.0013" not "$0.00"). Reuse it for any per-run money display.

## What Doesn't Work

- **2026-06-18** ‚Äî `pnpm db:seed` creates reviews/runs for the demo PRs but ZERO findings (the `findings` table is empty after a fresh seed). So the new PR-list FINDINGS column and the Agent-runs timeline findings both render their EMPTY states locally ("‚Äî" / "0 finding(s)") ‚Äî there's nothing to screenshot out of the box. To populate the findings UI you must run a real review (needs a non-empty `OPENROUTER_API_KEY` in `server/.env`; the 3 default agents use `deepseek/deepseek-v4-flash` via OpenRouter ‚Äî all keys ship empty). Inserting demo `findings` rows directly is blocked by the harness write-guard. Evidence: `GET /pulls/:id/reviews` returns `findings: []` for seeded PR #482.

## Codebase Patterns

- **2026-06-14** ‚Äî Cross-route shared components live in `src/components/<Name>/` with an `index.ts` barrel, imported via `@/components/<Name>` (e.g. `RunCostBadge`, `diff-viewer`). Vendored UI primitives (`Badge`, `CircularScore`) live in `src/vendor/ui` under `@devdigest/ui` ‚Äî different home. Evidence: `client/src/components/RunCostBadge/`.
- **2026-06-14** ‚Äî The PR-list table is driven by two parallel constants that MUST stay length-aligned: `COLUMN_KEYS` (header keys + order) and `GRID` (CSS grid-template tracks). Adding a column = add to both AND render a matching cell in `PRRow.tsx`, else header/cells misalign silently. Evidence: `client/src/app/repos/[repoId]/pulls/constants.ts`.
- **2026-06-14** ‚Äî i18n has only the `en` locale (`client/messages/en/`); new UI strings need a key under the right namespace file (e.g. `prReview.json`, `runs.json`) read via `useTranslations("<ns>")`. A missing key renders the raw key, not an error.
- **2026-06-18** ‚Äî Findings on the PR detail page are NOT a flat list: `FindingsTab` renders one `ReviewRunAccordion` per `ReviewRecord`, each owning its own `FindingsPanel`. A PR-level finding filter/counter must aggregate over `runs.flatMap(r => r.findings)` in `FindingsTab` and be threaded down as a prop (accordion ‚Üí panel). `visibleFindings()` in `FindingsPanel/helpers.ts` is the single choke point for finding filtering+severity sorting ‚Äî extend it (it already takes `hideLow`, now `severity?`) rather than filtering inside the component. Evidence: `client/src/app/repos/[repoId]/pulls/[number]/_components/FindingsPanel/helpers.ts`.
- **2026-06-18** ‚Äî The PR-list endpoint (`GET /repos/:id/pulls` ‚Üí `PrMeta[]`) DELIBERATELY omits per-severity finding counts ("findings live on the PR detail page" ‚Äî `server/src/modules/pulls/routes.ts:114`), and `PrMeta` lives in the do-not-touch `server/src/vendor/shared/`. To surface findings on the list WITHOUT backend changes: the list response already includes each PR's internal `id` (`routes.ts:176`), so fetch findings client-side per reviewed PR (`score != null`) with `useQueries` keyed `["reviews", prId]` ‚Äî the SAME key `usePrReviews` uses, so cache is shared with the PR detail page both directions. See `usePrFindingsForList` in `client/src/lib/hooks/reviews.ts`. The timeline (Agent runs) needs no fetch: `FindingsTab` already has `runs: ReviewRecord[]`, joinable to `prRuns: RunSummary[]` by `run_id`.

- **2026-06-27** ‚Äî Adding a tab to the agent/skill editor takes THREE edits in lock-step or the tab is invisible/dead: (1) the editor's `constants.ts` `TABS` (renders the tab button), (2) the route page's `VALID_TABS` (else `?tab=x` silently falls back to `config`), and (3) the editor component's body switch ‚Äî the shell previously hard-rendered `ConfigTab` regardless of `tab`, so a new tab needs an explicit `{tab === "x" && <XTab/>}`. Evidence: `client/src/app/agents/[id]/_components/AgentEditor/{AgentEditor,constants}.tsx` + `agents/[id]/page.tsx` (skills editor mirrors this at `app/skills/[id]/`).
- **2026-06-27** ‚Äî The sidebar nav lives in `src/vendor/ui/nav.ts` (`NAV` array) even though it's "vendored" ‚Äî the app's OWN items (`skills`, `agents`) are defined there, so adding a feature nav item means editing that file (there is no app-level extension point). A new `{ key, label, icon, href, gKey }` entry needs NO further wiring: `activeKeyFor()` in `app-shell/helpers.ts` derives the active item from the path and the `g`-shortcut handler iterates `NAV` automatically. Use `:repoId` in the `href` for repo-scoped pages (`resolveHref` substitutes the active repo) ‚Äî a global-section item can still point at a repo-scoped route. Evidence: `client/src/vendor/ui/nav.ts`, `client/src/components/app-shell/helpers.ts` (`activeKeyFor`).

- **2026-06-28** ‚Äî Smart Diff's per-line/-file finding overlay must read the LATEST review's findings (`reviews[0].findings`, reviews are newest-first), NOT the page's `allFindings` (which `flatMap`s across ALL runs). The server's `GET /pulls/:id/smart-diff` composes `finding_lines` from the single latest review only, so using `allFindings` would badge lines from stale/older runs that the server never included ‚Äî the diff overlay and the server's grouping would disagree. Evidence: `client/src/app/repos/[repoId]/pulls/[number]/_components/DiffTab/DiffTab.tsx` (`latestFindings = reviews?.[0]?.findings`), server `server/src/modules/pulls/routes.ts` smart-diff route (latest-review query).
- **2026-06-28** ‚Äî To deep-link from one tab to a specific item on another WITHOUT a reload, mirror the established `?trace=` pattern: a Smart Diff badge click calls `setParams({ tab: "findings", findingId: id })` ‚Äî note the COMBINED setter (one `router.replace` from a single `URLSearchParams`); two sequential `setParam` calls race because each rebuilds from the now-stale `search`. The target then cascades `targetFindingId` down `FindingsTab ‚Üí ReviewRunAccordion ‚Üí FindingsPanel ‚Üí FindingCard`; the accordion opens if it owns the finding, `FindingsPanel` sets `defaultExpanded` only on the match, and `FindingCard` self-scrolls via a ref + `scrollMarginTop` (`isTarget` effect). No nonce needed here (unlike Timeline‚ÜíReviewRuns) because the tab switch remounts `FindingsTab` fresh each click. Evidence: `client/.../page.tsx` (`setParams`, `targetFindingId`), `FindingCard.tsx` (`isTarget` effect).
- **2026-06-28** ‚Äî To overlay annotations on the shared diff viewer without forking it, thread OPTIONAL props through `FileCard ‚Üí CodeLine` (`lineFindings?: Map<newLineNo, Finding>`, `onFindingClick`, `defaultOpen`, `headerExtra`). Undefined in the plain `DiffViewer` path ‚Üí zero behaviour change there; `SmartDiffViewer` populates them. Match findings to a line by `ln.newNo` (the NEW-file line; `start_line` is post-change). `FileCard`'s open state defaults to a size heuristic ‚Äî pass `defaultOpen` to force it (Smart Diff collapses the boilerplate group with `defaultOpen={false}`). Evidence: `client/src/components/diff-viewer/{FileCard,CodeLine}/*.tsx`, `client/src/components/SmartDiffViewer/SmartDiffViewer.tsx`.

## Tool & Library Notes

- **2026-06-28** ‚Äî `vendor/ui`'s `Severity` token type has FOUR values (`CRITICAL|WARNING|SUGGESTION|INFO`) ‚Äî broader than the shared findings contract `Severity` (three; no `INFO`). A `Record<Severity, number>` keyed by the `@devdigest/ui` type therefore MUST include `INFO` or tsc fails (`TS2741`), even though review findings never use it. Evidence: `client/src/vendor/ui/primitives/tokens.ts` vs `vendor/shared/contracts/findings.ts`, `client/src/components/SmartDiffViewer/constants.ts` (`SEVERITY_RANK`).
- **2026-06-18** ‚Äî There is NO Tooltip/Popover/HoverCard primitive in `vendor/ui` (only `Modal`/`Drawer`, which are too heavy for inline hovers). For hover popovers that must escape table/row `overflow`, use the shared `HoverPopover` (`client/src/components/HoverPopover/`): it renders content via `react-dom` `createPortal` to `document.body`, positioned from the trigger's `getBoundingClientRect()` (fixed, viewport-clamped, flips above when the space below is cramped). Evidence: `client/src/components/HoverPopover/HoverPopover.tsx`.
- **2026-06-18** ‚Äî Verifying client data locally: the dev API (`:3001`) is reachable UNAUTHENTICATED and resolves a default workspace, so `curl` works directly ‚Äî `GET /repos`, `GET /repos/:id/pulls` (returns each PR's `id` + `score`), `GET /pulls/:id/reviews` (findings). The local Postgres container is `devdigest-postgres` with role/db `devdigest`/`devdigest` (NOT `postgres`). `./scripts/dev.sh` reuses an already-running Postgres but the API/web boot will `EADDRINUSE` if `:3001`/`:3000` are already up ‚Äî check `lsof -iTCP:3000` before relaunching.
